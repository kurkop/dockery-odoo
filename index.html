Usage Scenarios

There are two main usage scenarios.


Client Project


Characteristics:

  • Module dependencies are vendored in vendor/<ORG>/<REPO>/<MODULE>1.
  • src is used for custom development.
  • It is best practice to version control configuration and meta-data.
  • Such custom_config module satisfies the min. module condition.
  • Uses the migration controller for orchestrated module upgrades.

1 Use sparse-checkout to only get the modules you need.

Odoo Dev Team


Characteristics:

  • Stable repos are vendored in vendor/<MYORG>/<REPO>.
  • Module dependencies, if any, same as for Client Projects.
  • src is used for pre-release module development.
  • src is a shared workdir on the team’s remote repo. 1
  • The team uses feature branches to develop new modules.
  • After beta acceptance, the module is published in a vendored repo. 2
  • Further maintenance is done on that target repo.
  • Uses the migration controller for migration script validation.

1 Nice: src modules override their namesakes in vendored repos, as they are placed into .../090, the last (-> first) addon path.

2 Use “Initial commit” to squash development history or git subtree commands to preserve it or, to get fancy, adopt a git subrepo workflow to keep disconnected histories in sync during beta phase.

Remember:

  • Best use a git submodule workflow for your vendored repositories.
  • For sparse-checkout on submodules, start with this stackoverflow answer.
